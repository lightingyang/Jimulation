# Jimulation 1.0.9.3 Changelog

**发布日期：** 2026-03-27

---

## 后端

- 预处理开卷接口 `POST /api/v1/coil_uncoiling_estimate` 新增独立 `alerts` 输出
- 当某个批次在 `day 1 ~ deadline` 的可排窗口内即使拆分并开到夜班仍无法完成时，会单独输出产能预警
- 预警维度按“同截止日 + 同规格 + 同效率节拍”聚合
- `schedule.day` 改为直接返回输入窗口的原始排产日编号，不再重新编号
- 输出 `schedule` 固定覆盖 `1 ~ total_days`，空天保留为 0 任务日
- 告警字段不再暴露负数排产日，改为返回业务化的 `required_advance_days` / `required_advance_display`
- 告警新增 `overflow_time_display`，会按“天 / 小时 / 分钟”层级格式化缺口时长
- 预警统一以前置“窗口产能预检”为主，避免与排程后外溢预警重复
- `summary` 新增：
  - `capacity_alert_count`
  - `capacity_alert_sheets`
  - `capacity_alert_time_minutes`

## 排程逻辑

- 同截止日、同规格、同效率节拍的需求先合并成内部批次
- 批次排序从“同 deadline 下优先长批次”调整为“优先窗口最紧的批次”
- 窗口紧张度按“批次总工时 / 自身可排窗口总分钟数”衡量，负荷越高越优先
- 仍然优先整批贴近红线排产
- 先排白班，再扩到夜班
- 只有整批单日放不下时，才拆分到更早日期
- 若仍无法在可排窗口内完成，则继续排到窗口外，同时输出预警

## 2026-03-27 Optimization Notes

- The backend now builds uncoiling batches once per request and reuses them for both precheck alerts and scheduling.
- Capacity alerts are generated by the window precheck path only, so duplicate warnings from post-scheduling overflow are removed.
- Scheduling no longer allocates synthetic `day <= 0` rows. It schedules only inside `1..total_days`, and any unmet work is explained by alerts.
- Within the same deadline, precheck attribution follows the same tight-window-first priority as scheduling for more stable results.
